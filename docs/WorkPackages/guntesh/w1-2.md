## Database Entities:

### Library Repository Access:

- Organization - A certain library organization under a specific name
    - Object_id
    - Name
    - Active/Inactive
- Library Branch - Each  branch of an organization
    - id (Object_id) – Primary Key
    - organization_id (Object_id) – Reference to Organization
    - name (String)
    - address (String)
    - location (GeoJSON Point)
    - librarian (String)
    - serviceRadiusKm (Number)

### Book Repository Access:

- Book (under book module iun service layer)
    - id (ObjectId) – Primary Key
    - title (String)
    - author (String)
    - isbn (String)
    - genre (Array of Strings)
    - language (String)
    - Age rating
    - Collection Name (if any)
- BookCopy (Under circulation module in service layer)
    - id (ObjectId) – Primary Key
    - bookId (ObjectId) – Reference to Book
    - branchId (ObjectId) – Reference to Library Branch
    - barcode (String, Unique)
    - status (AVAILABLE / ISSUED / LOST / DAMAGED)
    - condition (GOOD / FAIR / POOR)
    - lastIssuedAt (Date)
    - lastReturnedAt (Date)’

### User Repository Access:

- User (Embedded Profiles)
    - id (ObjectId) – Primary Key
    - email (String, Unique)
    - phone (String)
    - status (ACTIVE / BLOCKED)
    - profiles (Embedded Array)
    - Profile Structure:
        - profileId (ObjectId)
        - name (String)
        - accountType (PARENT / CHILD)
        - ageGroup (String)
- Auth
    - email (references User - email)
    - password (hashed with JWT_SECRET_KEY)

### Payment Repository Access

- Payments
    - transaction_id
    - issue_id
- Penalties
    - issue_id - references issue_id from Issues
    - dueDate (Date) - references dueDate from Issues
    - returnDate (Date)
    - fineAmount (Number)
    - fineStatus (NONE / PENDING / PAID)

### Issue Repository Access

- Issues
    - issue_id
    - profileId (ObjectId)
    - copyId (ObjectId)
    - issueDate (Date)
    - dueDate (Date) - NULL if a book is bought and not issued/rented
    - status (ISSUED / RETURNED / OVERDUE)
- Deliveries
    - id (ObjectId) – Primary Key
    - transactionId (ObjectId)
    - branchId (ObjectId)
    - deliveryAddress (String)
    - scheduledAt (Date)
    - deliveredAt (Date) - NULL IF NOT DELIVERED
    - status (SCHEDULED / OUT FOR DELIVERY / DELIVERED / FAILED)

### Request Response Flow

# Issuing a Physical Book

### Step 1: Authentication

- Validate JWT
- Extract user role
- Ensure role = Parent

### Step 2: Book Validation

- Check book exists
- Check format = physical
- Check inventory > 0

### Step 3: Delivery Eligibility Check

- Fetch user coordinates
- Fetch library coordinates
- Run Haversine formula
- If distance > 8km → reject

### Step 4: Business Logic

- Create issue record
- Set:
    - issue_date = today
    - due_date = today + 14 days (example)
- Decrement inventory count

**Transaction must be atomic.**

### Step 5: Response

- Success Response
- Possible Error Responses
    - Inventory empty
    - Outside Delivery Radius
    - Transaction Failed

# Late Penalty Generation

### Step 1: Fetch Overdue Issues

Query:

- return_date IS NULL
- current_date > due_date

### Step 2: Calculate Fine

Example logic:

```
overdue_days = current_date - due_date

if overdue_days > 0:
    fine = overdue_days × 10

```

Optional:

- Grace period = 2 days

### Step 3: Update Records

- Update issue record
- Create payment record
- Mark user as "Fine Pending"

### Step 4: Send Notification

(Email or in-app notification)

# AI Recommendation

### Step 1: Authentication

Validate user

### Step 2: Fetch Book Metadata

- Retrieve books matching child age
- Retrieve book descriptions
- Filter by availability

### Step 3: Construct Prompt

Backend prepares structured prompt:

- Age
- Book catalog summary
- User query

### Step 4: Call AI API

Backend → External AI API

AI returns recommended titles.

### Step 5: Format Response
